# RentJaga

A clean React (Vite) starter that already includes Supabase authentication with:

- Email + password login/signup
- Phone OTP login
- Google OAuth login
- Session handling
- Protected post-login home screen
- Error boundary fallback screen

The goal of this project is to give you a practical auth foundation you can extend, not just a blank template.

## Stack

- React 19
- Vite 7
- Supabase JS v2
- ESLint 9

## Features

- Multi-auth UI with tabs (`Email`, `Phone`)
- Login/signup mode toggle for email auth
- OTP send + verify flow for phone auth
- Google OAuth button and redirect-based sign-in
- Auto session restore on refresh
- Auth state listener (`onAuthStateChange`)
- Logout support
- Graceful empty-env state when Supabase keys are missing

## Setup

1. Install dependencies:

```bash
npm install
```

2. Create a `.env` file in the project root:

```env
VITE_SUPABASE_URL=your_supabase_project_url
VITE_SUPABASE_ANON_KEY=your_supabase_anon_or_publishable_key
```

3. Start development server:

```bash
npm run dev
```

## Supabase Dashboard Configuration

Set these in your Supabase project before testing all auth methods.

### 1) Email Auth

- Go to `Authentication -> Providers -> Email`
- Ensure Email provider is enabled

### 2) Phone Auth (OTP)

- Go to `Authentication -> Providers -> Phone`
- Enable Phone provider
- Configure SMS provider credentials (Twilio or supported provider)
- Use E.164 phone format in UI (example: `+1234567890`)

### 3) Google OAuth

- Go to `Authentication -> Providers -> Google`
- Enable Google provider
- Add Google client ID and secret
- In Google Cloud Console, set authorized redirect URI to your Supabase callback URL

### 4) Redirect URLs

- In `Authentication -> URL Configuration`, include your local URL
- Typical local value: `http://localhost:5173`

## How Auth Works in This App

1. On app load, `App.jsx` checks for an existing session using `supabase.auth.getSession()`.
2. `onAuthStateChange` keeps React state synced with auth events.
3. If no session, auth UI is shown.
4. If session exists, app renders `HomePage`.
5. `Logout` calls `supabase.auth.signOut()` and returns user to auth screen.

## Environment Safety Behavior

If `VITE_SUPABASE_URL` or `VITE_SUPABASE_ANON_KEY` is missing:

- `supabaseClient.js` does not initialize client
- App shows a clear prompt telling you to add env vars
- Prevents runtime crashes due to undefined config

## Common Issues

### App says to add env vars even though `.env` exists

- Confirm variable names are exactly:
  - `VITE_SUPABASE_URL`
  - `VITE_SUPABASE_ANON_KEY`
- Restart dev server after editing `.env`

### Google login opens but does not return to app

- Check redirect URL in Supabase `URL Configuration`
- Check Google OAuth redirect URI configuration

### OTP is not received

- Verify phone provider is enabled in Supabase
- Verify SMS provider credentials are valid
- Verify phone input is in E.164 format

## Todo Feature (Supabase)

This project now includes a simple todo list in the logged-in `Features` section.

Create a `todos` table in Supabase SQL editor:

```sql
create table if not exists public.todos (
	id bigint generated by default as identity primary key,
	user_id uuid not null references auth.users(id) on delete cascade,
	title text not null,
	is_completed boolean not null default false,
	created_at timestamptz not null default now()
);

alter table public.todos enable row level security;

create policy "Users can read own todos"
on public.todos
for select
using (auth.uid() = user_id);

create policy "Users can insert own todos"
on public.todos
for insert
with check (auth.uid() = user_id);

create policy "Users can update own todos"
on public.todos
for update
using (auth.uid() = user_id);

create policy "Users can delete own todos"
on public.todos
for delete
using (auth.uid() = user_id);
```
